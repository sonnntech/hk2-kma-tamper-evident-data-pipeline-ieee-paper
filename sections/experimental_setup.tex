\section{Experimental Design and Setup}

\subsection{Pipeline Prototype}

The prototype implements a four-stage analytical pipeline: Source, Bronze, Silver, and Gold. Source data is generated synthetically using a fixed random seed to support reproducibility. Bronze stores ingested records, Silver applies normalization and transformation logic, and Gold stores aggregated outputs. Each stage emits evidence into the ledger.

For each stage, the ledger records a stage name, record count, schema hash, batch hash, block hash, previous block hash, transformation metadata, timestamp, and pipeline run identifier. The batch hash represents the canonicalized content of the stage output. The schema hash represents the structure of the stage output. The block hash represents the cryptographic digest of the block payload. The previous hash links the current block to the previous block in the same pipeline run.

The verification engine recomputes stage evidence and compares it against the stored ledger. A mismatch in content hash indicates data tampering. A mismatch in record count indicates insertion or deletion. A mismatch in block payload indicates ledger tampering. A mismatch in previous hash indicates a broken chain link.

\subsection{Tamper Scenarios}

The study evaluates six tamper scenarios and one clean scenario. The clean scenario, labeled NONE, is used to observe false positives. The tamper scenarios modify transaction values, delete records, insert fake records, modify stored batch hashes, modify transformation metadata, and modify previous-hash links.

\subsection{Procedure}

The experiment follows the procedure shown in Fig.~\ref{fig:procedure}. First, the environment and Delta tables are initialized. Second, synthetic source data is generated using a fixed seed. Third, the baseline and treatment pipeline executions are performed. Fourth, the verification engine validates the clean run. Fifth, a tamper scenario is applied. Sixth, verification is repeated to measure detection and traceability. Seventh, the baseline state is reset before the next scenario. Finally, benchmark runs are executed for multiple record counts.
